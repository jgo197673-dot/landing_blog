# AUBE 프로젝트 트러블슈팅 및 장애 해결 기록 (Troubleshooting & Resolution Log)

본 문서는 AUBE 외국인 고객 응대 웹앱 프로젝트 개발 및 QA 테스트 과정에서 직면했던 주요 장애 사항, 버그, 오동작 원인과 이를 해결해 나간 전체 트러블슈팅 과정을 상세 기록한 로그 문서입니다.

---

## 1. Vercel 배포 시 Gemini API Key 미설정 및 에러 대응

### 🚨 문제 증상
* 플로팅 챗봇 버튼과 챗봇 창은 정상적으로 렌더링되지만, 사용자가 어떠한 질문을 입력해도 대화창에 `"죄송합니다. 일시적인 오류가 발생했습니다. 잠시 후 다시 시도해주세요."`라는 안내 텍스트만 고정 노출되는 현상 발생.

### 🔍 원인 분석
1. **서버(API) 측 원인**: Vercel 웹 대시보드 환경변수(Environment Variables)에 챗봇의 핵심 동력인 `GEMINI_API_KEY`가 실제로 등록되어 있지 않았습니다. 이에 따라 서버리스 함수 `api/chat.js`가 API 키를 식별하지 못해 백엔드 통신 오류가 유발되었습니다.
2. **구글 API 키 형식 검증**: 일반적으로 구글 API 키는 `AIzaSy...` 형식을 사용하지만, 구글 클라우드 및 최신 서비스 인증 규격에 따라 **`AQ.Ab8...`** 형식의 API 키도 유효하게 사용됨을 최종 확인했습니다.

### 🛠️ 해결 과정
* Vercel 프로젝트의 **Settings ➜ Environment Variables**로 이동하여 실제 발급받은 `GEMINI_API_KEY` 값(형식: `AQ.Ab8...`)을 등록한 후, 프로젝트를 재배포(Redeploy)하여 Gemini AI 챗봇이 백엔드와 정상 통신하도록 복구했습니다.

## 2. API 예외 응답(5xx) 수신 시 클라이언트 예외 처리 오동작

### 🚨 문제 증상
* 서버리스 함수(`api/chat.js`)가 API 키 미인식 또는 외부 API 장애 시에 자체적으로 예외를 포착하여 대체 텍스트(예: "정확한 내용은 매장으로 직접 문의해주세요.")와 함께 연락처 카드를 띄우도록 설계되었으나, 실제 화면에는 여전히 범용 에러 문구(`"죄송합니다. 일시적인 오류..."`)만 계속 표출되는 현상.

### 🔍 원인 분석
* 기존 클라이언트 스크립트([chatbot.js](file:///c:/Users/PC/Desktop/subject/AUBE/chatbot.js))의 `fetch` 수신부 코드가 HTTP 상태코드가 2xx가 아닌 경우(예: 502 Bad Gateway) `response.ok`가 `false`가 됨에 따라 무조건 `catch` 예외 처리 블록으로 이동하도록 설계되어 있었습니다.
* 이 때문에 백엔드가 정상적으로 가공하여 전송한 폴백용 JSON 데이터(`answer`와 `needsContact: true` 플래그)가 통째로 묵살되는 문제가 발견되었습니다.

### 🛠️ 해결 과정
* `chatbot.js` 내의 `fetch` 통신 영역을 보완하여, HTTP 응답 상태가 실패(`!response.ok`)이더라도 응답 데이터가 유효한 JSON 규격이고 그 내부에 대체 답변(`data.answer` 또는 `data.reply`)이 포함되어 있다면 정상 렌더링을 진행하도록 분기 로직을 수정했습니다.
* 이로써 서버 장애 또는 키 만료 시에도 먹통이 되지 않고 연락처 카드(Contact Card)가 안정적으로 노출되도록 강건화(Robustness)를 달성했습니다.

## 3. 다국어 설정 전환 시 무한 상태 갱신 루프 및 브라우저 마비 현상

### 🚨 문제 증상
* 메인 웹사이트 상단에서 언어(한국어, English, 中文)를 연속적으로 전환하거나 변경 버튼 클릭 시 브라우저가 일시적으로 마비되거나 **"Minified React error #185 (Maximum update depth exceeded)"** 계열의 리소스 고갈 에러가 보고되는 현상.

### 🔍 원인 분석
* 메인 웹페이지의 언어 변경 함수(`changePageLang`, `setLang`)와 챗봇의 언어 변경 함수(`changeChatLang`, `window.changeChatbotLanguage`)를 동기화하는 구조에서, **현재 설정 언어와 변경 요청 언어가 동일한 경우**에 대한 리턴 예외 처리가 부재했습니다.
* 이로 인해 불필요한 DOM 상태가 중복 갱신되었으며, 특히 Vercel Insights/Analytics 등 전역 DOM의 어트리뷰트(HTML `lang` 등) 변동을 관찰하는 외부 트래커 라이브러리가 존재할 경우 상태 변경이 재귀적으로 맞물리며 무한 업데이트 루프에 빠지는 취약점이 식별되었습니다.

### 🛠️ 해결 과정
1. **동일 값 차단**: `changePageLang()`, `setLang()`, `changeChatLang()` 모든 다국어 변경 함수의 진입부에 변경하려는 언어가 현재 설정된 언어와 동일하다면 렌더링 및 동기화를 하지 않고 즉시 리턴하는 코드를 심었습니다.
   ```javascript
   if (lang === currentLang) return; // 동일 언어 요청 시 즉시 차단
   ```
2. **재진입 가드 플래그**: 동기화 시점에 중복 실행을 막는 락(Lock) 메커니즘인 `isLangSyncing` 플래그를 도입하여 동기화 작업이 완벽하게 완료될 때까지 추가 갱신 요청이 들어오지 못하게 통제했습니다.
   ```javascript
   if (isLangSyncing) return;
   isLangSyncing = true;
   try {
       // 동기화 및 렌더링 처리
   } finally {
       isLangSyncing = false;
   }
   ```
* 결과적으로 React 모니터링 라이브러리 및 브라우저 엔진과의 충돌을 완벽히 소거하여 안정적인 싱크 UX를 구현했습니다.

## 4. GA4 네이버 지도 클릭 시 `wechat_click` 이벤트 오집계 버그

### 🚨 문제 증상
* GA4 실시간 모니터링 검수 결과, 사용자가 매장 길 찾기(네이버 지도 링크) 버튼을 클릭할 때 실제 수집되는 이벤트가 `map_click` 또는 `location_click`이 아니라 엉뚱하게 **`wechat_click`**으로 오집계되어 마케팅 분석 데이터를 교란하는 현상.

### 🔍 원인 분석
* [notice.html](file:///c:/Users/PC/Desktop/subject/AUBE/notice.html) 하단에 이식된 GA4 클릭 이벤트 리스너 내 분기 조건 중, `href.includes('map.naver.com')` (네이버 지도)에 매칭되는 실행 결과가 이전 코드의 복사/붙여넣기 실수로 인해 `wechat_click` 이벤트를 송출하도록 코딩되어 있었습니다.

### 🛠️ 해결 과정
1. **이벤트명 교정**: 지도 링크를 클릭했을 때는 원래 의도했던 **`map_click`** 이벤트가 전송되도록 코드를 교정했습니다.
2. **위챗 클릭 추적 보강**: 기존에 단순 링크 감지만으로는 추적할 수 없었던 실제 WeChat QR 코드 영역(메인 페이지의 `.wechat-qr-area img`, 안내문의 `.wechat-qr-box img`, 챗봇 내부의 `.contact-card-qr img`)에 마우스 클릭 및 모바일 터치가 들어올 때 실질적인 **`wechat_click`** GA4 이벤트가 정상 카운트되도록 이벤트 리스너를 결합 및 신설하여 해결했습니다.

## 5. Gemini AI의 JSON 마크다운 펜스 반환에 따른 파싱 크래시

### 🚨 문제 증상
* AI가 답변할 때 정상적인 JSON 텍스트 대신 마크다운 문법인 ` ```json \n { ... } \n ``` `과 같이 코드 펜스(Fences) 기호를 씌워 리턴할 때 백엔드 `JSON.parse`가 문자열 형식을 해석하지 못하고 `SyntaxError` 예외를 내며 500 오류를 발생시키는 현상.

### 🔍 원인 분석
* API 호출 명세에 JSON 타입 출력을 제약해 두더라도 LLM 모델의 가중치나 내부 가드레일에 의해 코드 블록 문법이 임의로 결합되어 리턴되는 경우가 있으며, 백엔드 파서(`api/chat.js`)에 이에 대한 예외 처리가 마련되어 있지 않았습니다.

### 🛠️ 해결 과정
* `api/chat.js` 서버리스 함수에 응답 문자열의 마크다운 펜스 구문(시작 부분의 ` ```json `, ` ``` ` 및 끝부분의 ` ``` `)을 정규식 및 문자열 교체로 사전에 깔끔하게 발라내고 정제하는 코드를 추가하여 JSON 파싱 성공률을 100%로 고도화했습니다.
